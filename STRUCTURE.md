# Структура Laravel-проекту

## Огляд кореневої структури

### `/app`
Ядро застосунку, де знаходиться основна бізнес-логіка CRM. Містить:
- **Models/** - моделі Eloquent для роботи з таблицями (Client, Invoice, Service тощо)
- **Http/Controllers/** - контролери, які обробляють HTTP-запити
- **Http/Middleware/** - проміжне ПЗ для фільтрації запитів (авторизація, CORS тощо)
- **Providers/** - сервіс-провайдери для реєстрації компонентів
- **Console/Commands/** - власні Artisan-команди

*Порівняння з Module 3:* Тут аналог папок `src/Controllers/`, `src/Models/`, `src/Services/` з чистого PHP.

---

### `/bootstrap`
Завантажувач фреймворку. Містить:
- `app.php` - ініціалізація та реєстрація сервісів
- `cache/` - кешовані файли маршрутів та конфігурацій

*Порівняння з Module 3:* Аналог `config/app.php` або `public/index.php` у чистому PHP.

---

### `/config`
Конфігураційні файли застосунку:
- `database.php` - налаштування підключення до БД (читає з .env)
- `cache.php` - конфігурація кешування
- `app.php` - основні налаштування програми
- `services.php` - налаштування зовнішніх сервісів

*Порівняння з Module 3:* У чистому PHP ми мали окремі файли:
- `config/database.php` - PDO конфігурація
- `config/redis.php` - Redis налаштування
У Laravel всі конфіги централізовані в одній папці та працюють через env().

---

### `/database`
Все, що пов'язано з БД:
- **migrations/** - файли міграцій для створення/зміни таблиць (clients, invoices, etc.)
- **seeders/** - сиди для заповнення БД тестовими даними
- **factories/** - фабрики для генерації даних

*Порівняння з Module 3:* У чистому PHP ми створювали таблиці вручну через SQL-скрипти. Laravel використовує міграції для version control БД.

---

### `/public`
Точка входу в застосунок:
- `index.php` - єдиний вхідний файл (Front Controller)
- **css/**, **js/**, **images/** - публічні ресурси

*Порівняння з Module 3:* У чистому PHP було багато точок входу:
- `public/clients.php`
- `public/invoices.php`
- `public/accounts.php`
У Laravel всі запити проходять через один файл `public/index.php`, а маршрутизація визначається в `routes/`.

---

### `/resources`
Некомпільовані ресурси:
- **views/** - Blade-шаблони для відображення HTML (аналог наших `.php` файлів у папці `public/`)
- **lang/** - файли локалізації
- **js/**/**sass/** - вихідні JS/CSS файли

*Порівняння з Module 3:* Шаблони з чистого PHP (як `invoices.php`, `invoice-detail.php`) тепер будуть у `resources/views/`.

---

### `/routes`
Визначення маршрутів:
- `web.php` - маршрути для веб-інтерфейсу (сесії, cookies)
- `api.php` - маршрути для API (токени, rate limiting)
- `console.php` - маршрути для Artisan-команд
- `channels.php` - маршрути для WebSocket

*Порівняння з Module 3:* Замість прямих звернень до файлів `public/invoices.php`, ми визначаємо маршрут:
```php
Route::get('/invoices', [InvoiceController::class, 'index']);
```

---

### `/storage`
Логи, кеш, завантажені файли:
- **app/** - завантажені користувачами файли
- **framework/** - кеш, сесії, скомпільовані шаблони
- **logs/** - лог-файли застосунку

*Порівняння з Module 3:* У чистому PHP ми зберігали логи вручну. Laravel автоматично логує помилки та події.

---

### `/tests`
Автоматичні тести:
- **Feature/** - функціональні тести (CRUD-операції, API)
- **Unit/** - модульні тести (окремі класи, методи)

*Порівняння з Module 3:* У чистому PHP ми не мали структурованого тестування. Laravel має вбудовану підтримку PHPUnit.

---

### `/vendor`
Залежності Composer. Не зберігається в Git, автоматично встановлюється при `composer install`.

## Особливості Laravel

1. **Front Controller**: Всі запити йдуть через `public/index.php`, що забезпечує єдину точку входу.

2. **Маршрутизація**: У Laravel маршрути визначаються декларативно, а не через фізичні файли.

3. **Конфігурація**: Змінні середовища (.env) відокремлені від коду, що безпечніше для продакшену.

4. **Artisan**: Вбудовані CLI-інструменти для генерації коду, міграцій, тестів.

5. **Eloquent ORM**: Замість ручних PDO-запитів - об'єктно-орієнтована робота з БД.

6. **Blade**: Шаблонізатор з наслідуванням та компонентами (замість чистого PHP у шаблонах).
