# Сценарій: Переказ коштів між рахунками

## Опис сценарію

Оператор CRM виконує переказ коштів з одного рахунку на інший.
Після успішного переказу потрібно:

- Відправити email обом клієнтам
- Записати подію в аудит-лог
- Оновити кеш статистики

---

## Кроки сценарію

| №  | Крок                   | Опис                                            |
|----|------------------------|-------------------------------------------------|
| 1  | Зняття коштів          | UPDATE балансу рахунку відправника              |
| 2  | Поповнення коштів      | UPDATE балансу рахунку отримувача               |
| 3  | Створення транзакції   | INSERT у таблицю transactions (для відправника) |
| 4  | Створення транзакції   | INSERT у таблицю transactions (для отримувача)  |
| 5  | Відправка email        | Відправнику: "Ваш переказ виконано"             |
| 6  | Відправка email        | Отримувачу: "На ваш рахунок надійшли кошти"     |
| 7  | Запис в аудит-лог      | Логування дії оператора                         |
| 8  | Виклик зовнішнього API | Відправка даних до бухгалтерської системи       |
| 9  | Оновлення кешу         | Cache::forget('crm:clients:list')               |
| 10 | Оновлення кешу         | Cache::forget('crm:dashboard:stats')            |

---

## Класифікація кроків

### Синхронно (HTTP-запит)

**Кроки, які виконуються синхронно:**

| Крок                                 | Причина                                        |
|--------------------------------------|------------------------------------------------|
| 1. Зняття коштів                     | Критична операція - без неї переказ неможливий |
| 2. Поповнення коштів                 | Критична операція - без неї переказ неможливий |
| 3. Створення транзакції (відправник) | Потрібна для історії та аудиту                 |
| 4. Створення транзакції (отримувач)  | Потрібна для історії та аудиту                 |

**Чому синхронно?**

> Ці кроки є критичними для коректності переказу.
> Користувач повинен отримати відповідь "Переказ успішно виконано" ТІЛЬКИ
> після того, як гроші реально списані та зараховані.
> Якщо будь-який з цих кроків впаде, транзакція має бути повністю відкочена за допомогою rollback команди.

---

### Асинхронно (Черга)

**Кроки, які виконуються в черзі:**

| Крок                           | Причина                                  |
|--------------------------------|------------------------------------------|
| 5. Відправка email відправнику | Не впливає на факт переказу              |
| 6. Відправка email отримувачу  | Не впливає на факт переказу              |
| 7. Запис в аудит-лог           | Може виконатись пізніше                  |
| 8. Виклик зовнішнього API      | Може бути повільним або недоступним      |
| 9. Оновлення кешу клієнтів     | Покращує продуктивність, але не критично |
| 10. Оновлення кешу статистики  | Покращує продуктивність, але не критично |

**Чому асинхронно?**

> Ці кроки не впливають на успішність самого переказу.
> Користувач не повинен чекати, поки відправляться email або
> оновиться кеш. Якщо якийсь із цих кроків займе 5 секунд або впаде,
> основний переказ все одно буде виконано.
> Відкладене виконання покращує час відповіді API
> (з 7-10 секунд до 1-2 секунд).

## Приклад коду

### Синхронна частина (в контролері/сервісі)

```php
public function transfer(TransferRequest $request): JsonResponse
{
    DB::transaction(function () use ($request) {
        // 1. Зняття коштів
        $fromAccount->balance -= $request->amount;
        $fromAccount->save();
        
        // 2. Поповнення коштів
        $toAccount->balance += $request->amount;
        $toAccount->save();
        
        // 3. Транзакція для відправника
        Transaction::create([
            'account_id' => $fromAccount->id,
            'amount' => -$request->amount,
            'type' => 'transfer_out',
            'status' => 'completed',
        ]);
        
        // 4. Транзакція для отримувача
        Transaction::create([
            'account_id' => $toAccount->id,
            'amount' => $request->amount,
            'type' => 'transfer_in',
            'status' => 'completed',
        ]);
    });
    
    // Асинхронні задачі (в чергу)
    SendTransferEmailToSender::dispatch($fromAccount, $request->amount);
    SendTransferEmailToReceiver::dispatch($toAccount, $request->amount);
    LogAuditEvent::dispatch($request->user(), 'transfer', $request->all());
    NotifyAccountingSystem::dispatch($fromAccount, $toAccount, $request->amount);
    ClearClientsCache::dispatch();
    ClearDashboardCache::dispatch();
    
    return response()->json([
        'success' => true,
        'message' => 'Переказ успішно виконано'
    ]);
}
```

## Сценарій 2: Створення рахунку-фактури (Invoice)

### Опис сценарію

Оператор CRM створює рахунок-фактуру для клієнта.
Після успішного збереження рахунку потрібно:

- Згенерувати PDF-файл рахунку
- Відправити email клієнту з PDF у вкладенні
- Записати подію в аудит-лог
- Оновити кеш списку рахунків

---

### Кроки сценарію

| # | Крок                     | Опис                                     |
|---|--------------------------|------------------------------------------|
| 1 | Збереження invoice       | INSERT у таблицю invoices                |
| 2 | Збереження invoice_items | INSERT у таблицю invoice_items           |
| 3 | Генерація PDF            | Створення PDF-файлу рахунку              |
| 4 | Відправка email          | Відправка email клієнту з PDF-вкладенням |
| 5 | Запис в аудит-лог        | Логування дії оператора                  |
| 6 | Оновлення кешу           | Cache::forget('crm:invoices:list')       |

---

### Класифікація кроків

#### Синхронно (HTTP-запит)

**Кроки, які виконуються синхронно:**

| Крок                        | Причина                                  |
|-----------------------------|------------------------------------------|
| 1. Збереження invoice       | Критично - без цього рахунок не створено |
| 2. Збереження invoice_items | Критично - без позицій рахунок неповний  |

**Чому синхронно?**

> Користувач повинен отримати відповідь "Рахунок створено" ТІЛЬКИ після того,
> як дані реально збережені в базі. Якщо будь-який з цих кроків впаде,
> транзакція має бути повністю відкочена за допомногою rollback команди.

---

#### Асинхронно (Черга)

**Кроки, які виконуються в черзі:**

| Крок                 | Причина                                  |
|----------------------|------------------------------------------|
| 3. Генерація PDF     | Може займати 1-3 секунди                 |
| 4. Відправка email   | Може займати 2-5 секунд                  |
| 5. Запис в аудит-лог | Не впливає на факт створення рахунку     |
| 6. Оновлення кешу    | Покращує продуктивність, але не критично |

**Чому асинхронно?**

> Користувач не повинен чекати, поки згенерується PDF і відправиться email.
> Це I/O операції, які можуть тривати кілька секунд.
> Головне, що рахунок вже збережено в БД.
> Відкладене виконання цих кроків не впливає на факт створення рахунку,
> але значно покращує час відповіді для користувача.

---

### Чому PDF та email - типові кандидати на чергу?

| Операція            | Час виконання | Чому в чергу                                           |
|---------------------|---------------|--------------------------------------------------------|
| **Генерація PDF**   | 1-3 секунди   | Залежить від CPU, розміру документа, кількості позицій |
| **Відправка email** | 2-5 секунд    | Залежить від SMTP сервера, мережі, розміру вкладення   |

**Основні причини:**

1. **Повільні I/O операції** - робота з диском (PDF) та мережею (email)
2. **Не критичні для відповіді** - користувач не чекає на них
3. **Можуть падати** - при помилці можна повторити автоматично
4. **Дорожчі за час** - 5 секунд очікування = поганий UX

### Приклад Job для генерації PDF

```php
<?php

namespace App\Jobs;

use App\Models\Invoice;
use Barryvdh\DomPDF\Facade\Pdf;
use Illuminate\Contracts\Queue\ShouldQueue;
use Illuminate\Foundation\Queue\Queueable;
use Illuminate\Support\Facades\Storage;

class GenerateInvoicePdf implements ShouldQueue
{
    use Queueable;

    public function __construct(public Invoice $invoice) {}

    public function handle(): void
    {
        $pdf = Pdf::loadView('pdf.invoice', ['invoice' => $this->invoice]);
        
        Storage::put("invoices/invoice_{$this->invoice->id}.pdf", $pdf->output());
        
        // Оновлюємо шлях до PDF в БД
        $this->invoice->update([
            'pdf_path' => "invoices/invoice_{$this->invoice->id}.pdf"
        ]);
    }
}
```

```php
<?php

namespace App\Jobs;

use App\Mail\InvoiceMail;
use Illuminate\Contracts\Queue\ShouldQueue;
use Illuminate\Foundation\Queue\Queueable;
use Illuminate\Support\Facades\Mail;

class SendInvoiceEmail implements ShouldQueue
{
    use Queueable;

    public function __construct(public Invoice $invoice) {}

    public function handle(): void
    {
        $client = $this->invoice->client;
        
        Mail::to($client->email)->send(new InvoiceMail($this->invoice));
    }
}
```

## Сценарій для Уроку 5.2: Відправка email з підтвердженням переказу

### Назва Job

```php
SendTransferConfirmationJob
```

### Вхідні дані (конструктор)

```php
public function __construct(
    public int $transactionId,
    public string $fromAccountNumber,
    public string $toAccountNumber,
    public float $amount,
    public string $currency
) {}
```

### Що робить handle() - псевдокод

```php
public function handle(): void
{
    // 1. Знайти транзакцію в БД
    $transaction = Transaction::find($this->transactionId);
    
    // 2. Отримати дані відправника та отримувача
    $fromAccount = Account::with('clients')->find($transaction->from_account_id);
    $toAccount = Account::with('clients')->find($transaction->to_account_id);
    
    $sender = $fromAccount->client;
    $receiver = $toAccount->client;
    
    // 3. Сформувати текст листа для відправника
    $senderMessage = "Ваш переказ на суму {$this->amount} {$this->currency} 
                      на рахунок {$this->toAccountNumber} успішно виконано.";
    
    // 4. Сформувати текст листа для отримувача
    $receiverMessage = "На ваш рахунок {$this->toAccountNumber} 
                        надійшло {$this->amount} {$this->currency} 
                        від {$sender->full_name}.";
    
    // 5. Відправити email (логування)
    Log::info("Email to sender: {$sender->email} - {$senderMessage}");
    Log::info("Email to receiver: {$receiver->email} - {$receiverMessage}");
    
    // 6. (Опціонально) Зберегти факт відправки в БД
    NotificationLog::create([
        'transaction_id' => $this->transactionId,
        'sender_email' => $sender->email,
        'receiver_email' => $receiver->email,
        'sent_at' => now(),
    ]);
}
```

### Чому цей сценарій підходить для черги

| Характеристика | Чому в чергу |
|----------------|--------------|
| **Не критично для відповіді** | Користувач не чекає на email |
| **I/O операція** | Відправка email (SMTP) - повільна |
| **Може падати** | Email сервер може бути недоступний |
| **Retry механізм** | При помилці можна повторити |

### Де буде dispatch() в контролері

```php
class TransferController extends Controller
{
    public function store(TransferRequest $request): JsonResponse
    {
        // СИНХРОННО: збереження в БД (транзакція)
        DB::transaction(function () use ($request, &$transaction, &$fromAccount, &$toAccount) {
            // зняття коштів, поповнення, створення транзакцій...
        });
        
        // АСИНХРОННО: відправка email (в чергу)
        SendTransferConfirmationJob::dispatch(
            transactionId: $transaction->id,
            fromAccountNumber: $fromAccount->account_number,
            toAccountNumber: $toAccount->account_number,
            amount: $request->amount,
            currency: $request->currency
        );
        
        return response()->json([
            'success' => true,
            'message' => 'Переказ успішно виконано'
        ]);
    }
}
```
