# Сценарій: Переказ коштів між рахунками

## Опис сценарію

Оператор CRM виконує переказ коштів з одного рахунку на інший.
Після успішного переказу потрібно:

- Відправити email обом клієнтам
- Записати подію в аудит-лог
- Оновити кеш статистики

---

## Кроки сценарію

| №  | Крок                   | Опис                                            |
|----|------------------------|-------------------------------------------------|
| 1  | Зняття коштів          | UPDATE балансу рахунку відправника              |
| 2  | Поповнення коштів      | UPDATE балансу рахунку отримувача               |
| 3  | Створення транзакції   | INSERT у таблицю transactions (для відправника) |
| 4  | Створення транзакції   | INSERT у таблицю transactions (для отримувача)  |
| 5  | Відправка email        | Відправнику: "Ваш переказ виконано"             |
| 6  | Відправка email        | Отримувачу: "На ваш рахунок надійшли кошти"     |
| 7  | Запис в аудит-лог      | Логування дії оператора                         |
| 8  | Виклик зовнішнього API | Відправка даних до бухгалтерської системи       |
| 9  | Оновлення кешу         | Cache::forget('crm:clients:list')               |
| 10 | Оновлення кешу         | Cache::forget('crm:dashboard:stats')            |

---

## Класифікація кроків

### Синхронно (HTTP-запит)

**Кроки, які виконуються синхронно:**

| Крок                                 | Причина                                        |
|--------------------------------------|------------------------------------------------|
| 1. Зняття коштів                     | Критична операція - без неї переказ неможливий |
| 2. Поповнення коштів                 | Критична операція - без неї переказ неможливий |
| 3. Створення транзакції (відправник) | Потрібна для історії та аудиту                 |
| 4. Створення транзакції (отримувач)  | Потрібна для історії та аудиту                 |

**Чому синхронно?**

> Ці кроки є критичними для коректності переказу.
> Користувач повинен отримати відповідь "Переказ успішно виконано" ТІЛЬКИ
> після того, як гроші реально списані та зараховані.
> Якщо будь-який з цих кроків впаде, транзакція має бути повністю відкочена за допомогою rollback команди.

---

### Асинхронно (Черга)

**Кроки, які виконуються в черзі:**

| Крок                           | Причина                                  |
|--------------------------------|------------------------------------------|
| 5. Відправка email відправнику | Не впливає на факт переказу              |
| 6. Відправка email отримувачу  | Не впливає на факт переказу              |
| 7. Запис в аудит-лог           | Може виконатись пізніше                  |
| 8. Виклик зовнішнього API      | Може бути повільним або недоступним      |
| 9. Оновлення кешу клієнтів     | Покращує продуктивність, але не критично |
| 10. Оновлення кешу статистики  | Покращує продуктивність, але не критично |

**Чому асинхронно?**

> Ці кроки не впливають на успішність самого переказу.
> Користувач не повинен чекати, поки відправляться email або
> оновиться кеш. Якщо якийсь із цих кроків займе 5 секунд або впаде,
> основний переказ все одно буде виконано.
> Відкладене виконання покращує час відповіді API
> (з 7-10 секунд до 1-2 секунд).

## Приклад коду

### Синхронна частина (в контролері/сервісі)

```php
public function transfer(TransferRequest $request): JsonResponse
{
    DB::transaction(function () use ($request) {
        // 1. Зняття коштів
        $fromAccount->balance -= $request->amount;
        $fromAccount->save();
        
        // 2. Поповнення коштів
        $toAccount->balance += $request->amount;
        $toAccount->save();
        
        // 3. Транзакція для відправника
        Transaction::create([
            'account_id' => $fromAccount->id,
            'amount' => -$request->amount,
            'type' => 'transfer_out',
            'status' => 'completed',
        ]);
        
        // 4. Транзакція для отримувача
        Transaction::create([
            'account_id' => $toAccount->id,
            'amount' => $request->amount,
            'type' => 'transfer_in',
            'status' => 'completed',
        ]);
    });
    
    // Асинхронні задачі (в чергу)
    SendTransferEmailToSender::dispatch($fromAccount, $request->amount);
    SendTransferEmailToReceiver::dispatch($toAccount, $request->amount);
    LogAuditEvent::dispatch($request->user(), 'transfer', $request->all());
    NotifyAccountingSystem::dispatch($fromAccount, $toAccount, $request->amount);
    ClearClientsCache::dispatch();
    ClearDashboardCache::dispatch();
    
    return response()->json([
        'success' => true,
        'message' => 'Переказ успішно виконано'
    ]);
}
```

## Порівняння часу виконання

| Сценарій                  | Синхронно (сек) | Асинхронно (сек) |
|---------------------------|-----------------|------------------|
| Оновлення БД (4 операції) | 0.2             | 0.2              |
| Відправка email (2 шт)    | 4.0             | 0 (в чергу)      |
| Запис в аудит-лог         | 0.1             | 0 (в чергу)      |
| Виклик зовнішнього API    | 2.0             | 0 (в чергу)      |
| Оновлення кешу (2 шт)     | 0.1             | 0 (в чергу)      |
| **ВСЬОГО**                | **6.4 сек**     | **0.2 сек**      |

---

## Висновок

- **Синхронно:** тільки критичні операції, без яких переказ неможливий
- **Асинхронно:** всі допоміжні операції, які можуть виконатися пізніше
- **Результат:** користувач отримує відповідь за 0.2 секунди замість 6.4 секунд
